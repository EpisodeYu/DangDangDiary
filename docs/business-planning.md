## 1. 项目现状与近期路线

- 当当日记目前已完成大半。
- **近期一定会做的**：
  - 语音记录
  - UI 优化
- **尚未完全想清楚的**：AI 功能要不要做、要做哪些、怎么做。
- **当前完成度定位**：作为"个人 / 家庭使用的小玩具"已经没问题，通常可以归类为个人开发者作品。
- **距离商业化的差距**：见下一节盘点。

## 2. 走向商业化需要补的课

如果希望把这个项目上架商店并正式运营，至少需要覆盖以下维度。

### 2.1 硬件
- 更高规格的服务器部署后端
- 算力
- 存储
- 带宽

### 2.2 软件
- 数据安全
- 高并发
- 鲁棒性
- 监测报警系统
- 用户行为数据统计系统
- 功能性上的充值系统
- 其他当前还未意识到的部分

### 2.3 合规
- 用户输入的违规内容识别
- 用户协议
- ICP 备案
- 企业资质
- 微信平台、阿里云平台的认证

### 2.4 其他杂项
- 宣发营销
- 海外收款与报税
- 其他未预料到的运营性事务

> **个人判断**：在决定进一步投入之前，以上事项必须先想清楚，否则投入会失控。

## 3. 项目定位与自身约束

- **人力结构**：一人项目，从开发到运营只有"我 + AI agent"。
- **背景限制**：本人是嵌入式程序员，此前并未接触过前后端。
- **产品定位**：专注于**宠物记录**的 APP，边界要收紧，不扩散到泛生活记录。

## 4. 商业模式决策

### 4.1 排除的方案及理由
- **商城售卖宠物物品**：与定位不符，且涉及监管与商务对接问题。
- **广告**：不考虑。
- **买断制**：项目有后端，必然存在持续性成本投入，买断无法支撑长期运营。

### 4.2 选定方向
- **订阅制**为唯一合理方向。
- 订阅制内部再分两种子模式：
  1. 必须先交钱才能使用；
  2. 基础功能免费 + 会员解锁高级功能 / 放宽限制。

### 4.3 上架地区
- 北美
- 日本
- 中国

### 4.4 个人取舍
- 子模式 1（上来就收费）：对一个名不见经传的项目而言，基本无法获取用户。
- 子模式 2（免费 + 会员升级）：更合理的默认选择；但需要谨慎设计**免费用户的成本敞口**——哪怕付费转化率不足，免费存量也不能把成本压垮。

## 5. 会员权益设计草案

设计原则：既要压住免费用户的成本敞口，又不能让免费体验差到失去用户。

### 5.1 存储空间
- **免费用户**：必须设上限。云服务器存储昂贵，用户量一旦放大，仅凭免费转付费的转化率不足以覆盖成本。
- **免费额度**：取"合理值"——既不过度占用成本，也不显得太抠。
- **付费会员**：在定价基础上给出更高的额外存储空间。

### 5.2 原图存储 / 上传
- **初衷**：希望全员可存原图——宠物从小到大的照片是珍贵记忆，期望妥善保存，且随时能下载回本地，哪怕丢手机也能找回。
- **现实妥协**：如果全员原图，项目会变成高成本的存储密集型项目，对一人项目并不合适。
  - 例外：如果只打算自己和家人使用，才会考虑全员原图。
- **折中方案**：将"上传原图"作为付费会员的可选权益。

### 5.3 宠物档案分享上限
- **免费用户**：档案分享数量设上限。
- **付费会员**：可分享给更多人。
- **已识别的规避风险**：用户可能通过"把宠物档案平均分给多个账号再互相分享"来绕过单人上限。
- **对策方向**：限制的对象应是"**拥有 + 被分享**"档案的总数量，而不只是"主动分享出去的数量"。

### 5.4 AI 功能会员专享
- API 调用开销较大，天然适合作为付费门槛。
- 具体做哪些 AI 功能见下一节。

## 6. AI 功能取舍

### 6.1 要做
- **猫正脸照识别"忍痛"**：
  - 契合 APP 定位（宠物记录 + 精细化养护）。
  - 对精细化养宠用户价值明确。

### 6.2 候选（倾向做）
- **AI 一键生成宠物表情包**：
  - 复用我独有的"宠物图片数据库"优势（用户既有档案照可直接喂给 AI，无需临时上传）。
  - 与项目调性相容。

### 6.3 放弃
- **根据提示词生成某档案宠物的图片 / 视频**（通用生成类）：
  - **打不过通用 AI**：豆包、千问、Grok 等通用且免费的 AI 在这个场景下没有明显劣势。
  - **我的优势有限**：我只是 API 的调用者，唯一优势是"不用用户再上传图片的宠物图库"；但用户真正要玩通用生成，大概率直接去通用 AI 自己传图、自己写提示词。
  - **提示词维护成本高**：需要实时追踪"当下什么宠物 AI 图片/视频最火"并预设模板，即便全职投入也很难覆盖。
  - **调性不符**：追热点与"安静记录宠物"的项目基调不一致。

> 一句话结论：宠物图片数据库这个独家优势，用来做**表情包**比用来做**通用生成**更契合。

## 7. 卖点与目标用户

对于这种小项目，**能命中需求并卖出去比产品本身的精雕细琢更关键**；项目问题都可以再迭代。

### 7.1 纯粹宠物内容需求者
- 喜欢打开一个 APP 里面就是"我的宠物"，没有其他乱七八糟的东西。
- 不像系统相册什么都有，找一张猫的图片要翻半天。

### 7.2 精细化养宠用户
- 驱虫、梳毛、洗澡等按时按点做的事务需要记录，自己安心。
- 记录本身也是对宠物的珍贵回忆。
- "AI 识别忍痛"这类功能正对口。

### 7.3 长久存储需求者
- 纯粹看中长期稳定保存宠物资料的能力。

### 7.4 宠物档案分享（最重要的卖点）

这是区别于"又一个宠物记账 APP"的核心价值，按场景展开：

1. **家庭多人共养同一只 / 几只宠物**
   - 大家都会拍照，每次在群里分享麻烦，还占用群的信息量。
   - 也不可能把整个个人相册共享给家人。
   - 在当当日记里：共享一份宠物档案，每个人拍照后上传，大家都看得到；只含宠物内容，不必担心个人相册隐私问题。
   - 副产物：多人协作的驱虫、剪指甲等事务，记录后可静默告知其他成员，避免重复劳动。
2. **异地养宠**
   - 人在外地上学/工作，宠物留在老家。
   - 每天让父母拍照分享太麻烦；不主动分享又惦记。
   - 共享宠物档案后，父母记录即我可见，无需额外沟通。
3. **宠物送养 / 转让**
   - 因种种原因需要把宠物转交给下一个人。
   - 搭配后续的**宠物档案转让**功能：
     - 下一任主人可以无痛继承并继续分享这只宠物的"前半生"。
     - 送出去的前主人也能获知宠物的"后半生"。
     - 两人不必过多交流、不必侵占彼此生活和隐私，就能共享这些珍贵的数字记录。
   - 扩展场景：**宠物店**可以在宠物幼年时就开始记录，在卖给买家时附赠整份幼年记录。大多数人都没有见过自己宠物几个月大时的样子，这份记录价值很高。
4. **只读分享父母 / 朋友的宠物档案**
   - 搭配"只读权限"。
   - 围观别人家宠物的生活也是一种乐趣。

## 8. UI 与视觉调性

- **总基调**：高级但简约。
- **底线**：不能让人一眼觉得是小作坊产品而丧失信心——这对社交媒体传播也很关键。
- **主色**：黑白日记本配色。
- **背景元素**：宠物简笔画风图案。
- **条目呈现**：记录等页面中的条目可用**不同颜色的浅色背景**做区分，模拟荧光笔 / 便利贴的质感。

## 9. 待决策 / 明显缺口

以下问题是商业化绕不开、但目前尚未给出明确答案的点，先登记，不展开：

- 免费用户存储空间的**具体阈值**与会员扩容**梯度**
- 订阅**定价**及各地区（北美 / 日本 / 中国）差异
- ICP 备案 vs 海外先上架的**先后顺序与时间线**
- 企业资质的**筹备时间点**（一人主体 vs 注册公司）
- **监测报警系统** 与 **用户行为统计系统** 的最小可用方案
- **违规内容识别**的方案选型
- **海外支付 / 报税**主体与路径
- **宣发渠道**与**首批种子用户**的获取来源
- AI 功能的**调用成本模型**与会员定价的匹配
